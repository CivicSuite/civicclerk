# Plan: Deterministic capture ordering via capture_seq

**Date:** 2026-06-10
**Branch:** work/clerk-persistence-phase-1
**Bug:** All four CivicMeetings persistence repositories (`civicclerk/motion_vote.py`,
`civicclerk/minutes.py`, `civicclerk/public_archive.py`) order list reads by
`(created_at, id)`. `id` is a random uuid4, so rows inserted within the same
timestamp tick tiebreak randomly and insertion order is not preserved.
`tests/test_production_depth_motion_vote_persistence.py::test_corrections_are_append_only_and_listed_in_insertion_order`
flakes ~40%.

## Design decision: app-side monotonic allocation (not DB identity)

Allocate `capture_seq = COALESCE(MAX(capture_seq), 0) + 1` inside the same
insert transaction, under the existing `_chain_lock`.

Why not a Postgres `GENERATED BY DEFAULT AS IDENTITY` column:
- The repositories create mirror tables via `metadata.create_all` on SQLite.
  SQLAlchemy cannot render a non-PK identity column on SQLite, so DB identity
  would force dialect-divergent repository code (the exact thing the mirror
  tables exist to avoid).
- Every write to these tables already serializes through `_chain_lock`
  because the in-memory `AuditHashChain` requires a single writer process.
  MAX+1 inside the insert transaction is race-free under the same
  single-process consistency model the audit chain already mandates.
